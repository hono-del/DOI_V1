#### 1. 課題の詳細

```markdown
□ カスタマイズオプションの種類と数
  - 何種類のオプションがあるか？
  - どのような組み合わせパターンがあるか？
  - 営業マンはどこまで把握しているか？

□ 営業プロセスの現状
  - 顧客ヒアリング → 提案 → 見積の流れ
  - 各ステップにかかる時間
  - ボトルネックはどこか？

□ 具体的な困りごと
  - 最適な提案ができない（精度60%程度？）
  - 見積作成に時間がかかる（何時間？）
  - 受注機会の損失（どれくらい？）
  - 顧客からのクレーム・要望

□ 現在の対応方法
  - マニュアル参照
  - ベテラン営業マンに相談
  - 技術部門に問い合わせ
  - それぞれの課題は？
```

#### 2. 顧客とのPoC状況

```markdown
□ 顧客情報
  - 顧客名（言える範囲で）
  - 業種・規模
  - 関係性（既存 or 新規）

□ PoC合意内容
  - 何を検証するのか？
  - 期間・スケジュール
  - 成功基準（KPI）

□ 顧客の期待
  - どんな機能を求めているか？

□ 制約条件

  - データ提供の可否
```

```markdown
    （Excel、データベース、PDF、紙）

  - データ量（何製品、何オプション）
```

#### 4. 予算とリソース

```markdown
  - 予算の出所（部門予算 or プロジェクト予算）
  - 追加予算の可能性

□ 人的リソース
  - プロジェクトに割ける人員

□ スケジュール制約
```

## 💡 提案する共同推進の枠組み

### 基本コンセプト

```yaml
【提案の本質】
「2部門の異なる課題を、共通のAI基盤で同時解決する」

❌ 別々に開発:
  - 自動車: 300万円
  - 工作機械: 300万円
  - 合計: 600万円
  - 問題: 重複開発、ノウハウ分散

✅ 共同開発:
  - 共通基盤: 400万円（両部門で折半）
  - 個別開発: 各100万円
  - 合計: 600万円（同じ予算で2倍の成果）
```

### 技術アーキテクチャ案

```yaml
【共通基盤（70%）】
両部門で共同開発・共有

1. RAGコアシステム
   - ベクトルDB（Pinecone / Weaviate）
   - セマンティック検索エンジン
   - ハイブリッド検索（ベクトル + キーワード）
   - チャンキング最適化

2. LLM統合レイヤー
   - GPT-4 / Claude API連携
   - プロンプトエンジニアリング
   - コスト最適化
   - レスポンス時間管理

3. インテント検出エンジン
   - ユーザー意図の自動分類
   - 最適な回答形式の選択
   - コンテキスト理解

4. データ管理基盤
   - メタデータ構造化
   - バージョン管理
   - アクセス制御
   - 更新ワークフロー

5. 分析・モニタリング
   - 利用状況分析
   - 精度測定
   - コスト管理
   - ダッシュボード

【個別カスタマイズ（30%）】
各部門で独自開発

自動車部門:
- 取扱説明書データの最適化
- 状況認識アルゴリズム
  - 車両データ連携
  - 環境データ（天候、位置）
- プロアクティブ通知
- マルチモーダル配信
  - 音声、AR、動画

工作機械部門:
- 製品仕様データの構造化
- カスタマイズロジック
  - オプション組み合わせルール
  - 制約条件チェック
- 見積システム連携
- 営業支援UI
  - 顧客ヒアリング支援
  - 提案書生成
```

## 🤔 想定される懸念と対応

### 懸念1: 「技術が違いすぎるのでは？」

```yaml
懸念:
「自動車（B2C）と工作機械（B2B）は
 ユースケースが違いすぎて、
 共通化できないのでは？」

対応:
「確かにユースケースは異なりますが、
 技術的な本質は同じです」

共通点:
✅ 複雑な情報から最適解を見つける
✅ 自然言語での質問応答
✅ ユーザーの意図を理解する
✅ 最適な情報を提示する

相違点は「個別カスタマイズ」で対応:
- 自動車: 状況認識、プロアクティブ通知
- 工作機械: カスタマイズロジック、見積連携

共通基盤70%、個別30%で実現可能
```

### 懸念2: 「予算・リソースの分担が複雑では？」

```yaml
懸念:
「予算の出所が違う、
 リソースの調整が大変では？」

対応:
「シンプルな分担案を提案します」

推奨案:
- 全社R&D予算として600万円申請
- 各部門の予算負担なし
- 開発チームは共通
- 各部門はリエゾン1名のみ

メリット:
✅ 予算調整がシンプル
✅ 「全社戦略」として位置づけ
✅ 社長への説得力向上
```

### 懸念3: 「スケジュールが合わないのでは？」

```yaml
懸念:
「工作機械は顧客とのPoC期限がある。
 自動車のスケジュールと合わないのでは？」

対応:
「並行開発で対応します」

Month 1: 共通基盤開発（両部門共通）
Month 2: 個別カスタマイズ（並行開発）
Month 3: 並行検証（各部門で独立）

工作機械の期限に合わせて:
- Month 2で工作機械を優先
```

```yaml
✅ 具体的な数字を示す
```
