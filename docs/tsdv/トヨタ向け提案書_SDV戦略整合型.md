## 1. エグゼクティブサマリー

### 1.1 提案の核心

トヨタ様が掲げるSDV戦略の最重要目的「**交通事故ゼロ**」の実現に、弊社の**Human Safety Experience OS**で貢献いたします。

```
【SDVで事故がなくならない"最後の壁"】
トヨタ様がどれだけ高度なSDVを開発しても、
「人間の理解・判断・行動」の問題で事故は起きる
        ↓
【弊社の解決策】
「人とSDVの間の意味変換レイヤー」を提供
= Human Safety Experience OS
        ↓
【共創による価値】
車両（トヨタ様） × 情報体験（弊社）
= 「交通事故ゼロ」の実現
```

**弊社のキャッチフレーズ**:  
_"We translate SDV into human-safe experience."_  
（SDVを人間にとって安全な体験に翻訳する）

### 1.2 提案の戦略的位置づけ

```yaml
弊社が担う領域:
  「人とSDVの間の意味変換レイヤー」
  
  = OEMが内製しづらく、やると重くなる領域
  = しかし交通事故ゼロには不可欠な領域

具体的な責任:
  ❶ 機能理解 - 警告の意味を"納得できる形"で説明
  ❷ 判断支援 - 「何を優先すべきか」を明示
  ❸ 行動誘導 - 操作手順を"迷わない形"に分解
  ❹ 学習定着 - 同じミスを繰り返させない体験設計
  ❺ 責任説明 - 法的説明責任をUXとして成立させる
```

## 2. トヨタ様SDV戦略への理解

### 2.1 SDV戦略の核心理解

弊社は、トヨタ様のSDV戦略を以下のように理解しております。

#### トヨタ様の特徴的なアプローチ

```yaml
一般的なSDV:
  - エンターテインメント性重視
  - 収益化（アプリ/サービス）
  - 技術的先進性のアピール

トヨタ様のSDV:
  - 「交通事故ゼロ」という社会価値
  - 安全・安心の最大化
  - 車×インフラ×人の三位一体
  
結論:
  「単なるモノ売りから、
   社会安全の実現に貢献する
   持続的プラットフォームへ」
```

### 2.2 SDVがもたらす課題

SDVの特性上、以下の課題が想定されます。

| SDVの特性 | 生じる課題 | 弊社の解決策 |
|:--|:--|:--|
| **機能の継続的追加** | ユーザーが新機能を使いこなせない | 機能追加時の自動ガイド |
| **複雑化する操作** | 誤操作・事故のリスク増大 | 先回り警告・段階的ガイド |
| **多様なユーザー** | 習熟度に応じた説明が必要 | パーソナライズ機能 |
| **データ活用の必要性** | 利用実態の把握が困難 | データ還流システム |

## 3. 提案の背景と課題認識

### 3.1 自動車業界の現状課題

#### 課題1: 取扱説明書は読まれない

```yaml
現状:
  - 500ページの取説は実質的に読まれない
  - 機能活用率: わずか10%程度
  - ユーザーは多くの機能を知らないまま

影響:
  - 安全機能が使われない → 事故リスク
  - 便利機能が使われない → 満足度低下
  - 問い合わせ増加 → コスト増大
```

#### 課題2: コールセンター・販売店の負担

```yaml
コールセンター:
  - 月間問い合わせ: 5,000件（想定）
  - 年間コスト: 3,000万円
  - 内容: 基本操作40%、メンテナンス25%、トラブル20%
  
販売店:
  - 納車時説明: 1台2時間
  - 納車後問い合わせ対応: 月3回/台
  - 年間コスト: 1,260万円
  
削減可能性: 60-70%
```

#### 課題3: SDV時代の新たな課題

```yaml
従来車:
  - 買った時点で「完成」
  - 機能は固定
  - 取説も固定

SDV:
  - 継続的に「進化」
  - 機能が追加・変更される
  - 取説も更新が必要
  
問題:
  - 紙の取説では対応不可能
  - ユーザーは常に学習が必要
  - 新機能を使いこなせない
```

## 4. ソリューション概要

### 4.1 システムコンセプト：Human Safety Experience OS

```
【従来の取扱説明書の限界】
「探す」→「読む」→「理解する」→「実行する」
問題: ユーザーの能動的行動が前提
     → 事故が起きる4つの層（認知・判断・行動・継続）を解決できない

【弊社のHuman Safety Experience OS】
❶ 理解させる（認知の改善）
❷ 判断を補助する（判断の改善）
❸ 行動を誘導する（行動の改善）
❹ 学習を定着させる（継続の改善）
     ↓
事故を起こさせないために、
人の理解・判断・行動を"事前に"変えるOS
```

### 4.2 3レイヤーアーキテクチャ

弊社のシステムは、3つのレイヤーで「交通事故ゼロ」に貢献します。

#### レイヤー① Safety Knowledge Skeleton（知識OS）

```yaml
役割:
  取扱説明書DXの進化形
  = 「説明責任をSDVに載せるための外部脳」

機能:
  警告灯／ADAS／SDV機能を
  「人の判断単位」で再構造化

具体的な変換:
  【従来の取説】
  - 警告灯一覧（50個の羅列）
  - 機能説明（200ページの文書）
  - 操作手順（手順書）
  
  【知識OS】
  - 「この警告灯は何を意味するか」
  - 「今すぐすべきこと」（優先順位付き）
  - 「なぜそれが重要か」（理由の説明）
  - 「放置するとどうなるか」（結果の予測）
  ↓
  人の認知・判断・行動に直結する形に変換

法規・責任説明:
  - 法的に必要な情報を100%網羅
  - しかしUXとして破綻しない形で提示
  - デジタル記録で説明責任を証跡化
  - PL法・製造物責任への対応

技術基盤:
  - RAG最適化技術（詳細は6章）
  - セマンティック境界チャンキング
  - インテント別動的最適化
  - 回答精度90-95%
```

#### レイヤー② Contextual Safety Guidance（文脈型支援）

```yaml
役割:
  SDVと連動する"意味付けエンジン"
  = 状況を「今すべきこと」に翻訳

統合判断:
  状況（雨・夜・渋滞・初心者）
  × 車両状態（速度・燃料・警告灯）
  × 過去行動（よくする誤操作・習熟度）
  ↓
  「今、何に注意すべきか」
  「なぜそれが重要か」
  「具体的にどうすればいいか」
  を総合的に判断

データソース:
  【車両データ】
  - 速度・位置情報（GPS）
  - 警告灯の点灯状態
  - 走行距離・メンテナンス履歴
  - 使用している機能
  
  【環境データ】
  - 天候（雨・雪・霧）
  - 時間帯（朝・昼・夜）
  - 交通状況（渋滞・高速道路）
  - 季節（夏・冬）
  
  【ユーザーデータ】
  - 使用履歴・習熟度
  - スキルレベル（初心者・中級・上級）
  - よくする誤操作のパターン
  - 過去の理解度・判断速度

判断支援の例:
  【状況: 雨天 + 高速道路 + 初心者】
  
  ❌ 単なるAIチャット:
  「雨天時の運転方法は...（一般論の羅列）」
  
  ⭕ 弊社の文脈型支援:
  「高速道路で雨が強くなっています。
   
   今すべきこと（優先順位）:
   ① 車間距離を2倍に（最優先）
   ② VSC（横滑り防止）がONか確認
   ③ 速度を80km/h以下に
   
   なぜ重要か:
   雨天時は制動距離が1.5倍になります。
   初心者の方は特に車間距離の確保が重要です。
   
   結果予測:
   この対応で、ハイドロプレーニング事故の
   リスクを80%削減できます。」

重要な差別化:
  ❌ 「AIチャット」= 質問に答えるだけ
  ⭕ 「判断支援システム」= 体験設計された支援
  
  違い:
  - 状況を総合的に理解
  - 優先順位を明示
  - 理由を納得できる形で提示
  - 行動に直結する具体的指示
  - 結果予測で動機づけ
```

#### レイヤー③ Safety Experience Analytics（学習ループ）

```yaml
役割:
  事故ゼロに向けたUXログ基盤
  = 「SDV改善に直結する人間側データ」

収集データ:
  【❶ 認知の問題】

  - 優先順位が分からなかった場面
  
  【❸ 行動の問題】
  - 同じ誤操作が再発しているか
  - 正しい手順を覚えていないか
  - 操作の迷いが発生する機能
  - 緊急時の対応速度
  
  【❹ 継続の問題】

データ還流:

  【弊社システム改善へ】
  - 説明が分かりにくい箇所の改善

  - タイミング最適化
  - 学習効果の最大化

トヨタ様への独自価値:
  ハード・ソフトのログだけでは分からない
  「人間がどう理解・判断・行動したか」のデータ
  
  → 真の意味での「ユーザー中心設計」を実現
  → SDV開発の精度向上
  → 交通事故ゼロへの具体的な貢献
```

### 5.2 「交通事故ゼロ」への具体的貢献

#### 5.2.1 SDVで事故がなくならない"最後の壁"

トヨタ様がどれだけ高度なSDVを開発されても、**人間側の「情報体験の問題」**により事故は発生します。

```
【事故が起きる4つの層】

層          事故が起きる理由                 SDVの限界
──────────────────────────────────────────────────
❶ 認知     機能・警告の意味を理解していない    → ハードでは解決不可
❷ 判断     「今どうすべきか」が瞬時に分からない → AIだけでは不十分  
❸ 行動     正しい操作を迷う／遅れる           → 制御だけでは防げない
❹ 継続     学習・習熟が定着しない             → OTA更新だけでは不足

結論: これは「情報体験の問題」= 弊社の責任領域
```

#### 5.2.2 弊社の位置づけ：「Human Safety Experience OS」

```yaml
従来の取扱説明書DX（Before）:
  - 正しい情報を「探せば」ある
  - 事故後・警告後の説明が中心
  - 静的・一方向

SDV時代の体験OS（After）:
  - 事故を起こさせないために、
    人の理解・判断・行動を"事前に"変える
  - 動的・双方向・学習型

体験OSの役割（安全軸）:
  ❌ 説明する
  ⭕ 理解させる
  ⭕ 判断を補助する
  ⭕ 行動を誘導する
```
