# Phase 1: 7カテゴリ分類の定義

**作成日**: 2025年1月
**フェーズ**: Phase 1（2025-2026年）
**目的**: 2030年に取扱説明書をなくすための、7カテゴリ分類体系の詳細定義

---

## 1. 分類体系の概要

### 1.1 7カテゴリ分類の全体像

```
① コンテキスト依存操作（文脈理解 → プロアクティブ提示）
② リスク・危険・注意（Safetyモデル化）
③ 法規制・コンプライアンス・説明責任（Regulatory & Obligation）
④ メンテナンス・点検指示（予兆保全モデル）
⑤ スペック・仕様・固有値（IDパーソナライズ）
⑥ 状態異常・トラブルシュート（状態遷移モデル）
⑦ ライフスタイル・目的別利用（目的理解OS）
```

### 1.2 各カテゴリの提供方法マッピング

| カテゴリ | 主な提供方法 | 補完的提供方法 | 優先度 |
|---------|------------|--------------|--------|
| ① コンテキスト依存操作 | プロアクティブ通知 | オンデマンド検索 | 高 |
| ② リスク・危険・注意 | プロアクティブ通知 | デジタル記録 | 最高 |
| ③ 法規制・コンプライアンス | デジタル記録 | プロアクティブ通知 | 高 |
| ④ メンテナンス・点検指示 | プロアクティブ通知 | - | 高 |
| ⑤ スペック・仕様・固有値 | デジタル記録 | プロアクティブ通知 | 中 |
| ⑥ 状態異常・トラブルシュート | プロアクティブ通知 | - | 高 |
| ⑦ ライフスタイル・目的別利用 | プロアクティブ通知 | オンデマンド検索 | 中 |

## 2. カテゴリ①: コンテキスト依存操作（文脈理解 → プロアクティブ提示）

### 2.1 定義

**コンテキスト依存操作**とは、ユーザーの行動、環境、状況に応じて、適切なタイミングで操作ガイドを自動的に提供する情報である。

### 2.2 技術モデル

**文脈理解システム（Context Understanding System）**
- センサーデータ、位置情報、天気情報、時間情報を統合
- ユーザーの行動パターンを学習
- 適切なタイミングで自動提示

### 2.3 分類キーワード

#### 基本操作
- 警告灯、表示灯、計器、メーター、スピードメーター
- 燃料計、オドメーター、トリップメーター、外気温
- マルチインフォメーションディスプレイ、操作方法、使い方
- エアコン、ワイパー、給油、給油口、操作手順
- 設定、カスタマイズ、初期設定、HUD、ヘッドアップディスプレイ

#### 高度な操作
- カスタマイズ、設定変更、初期設定、ユーザーカスタマイズ
- 運転支援、LTA、PCS、RSA、エネルギーモニター
- ハイブリッドシステム、機能、システム

### 2.4 判断基準

✅ **このカテゴリに分類する**:
- 操作手順、使い方の説明
- 初回使用時、操作開始時に提供すべき情報
- 環境・状況に応じて提供内容が変わる情報
- センサーデータ、位置情報、天気情報と連動できる情報
- ユーザーの行動パターンで予測できる情報

❌ **このカテゴリに分類しない**:
- 危険警告（→ カテゴリ②）
- 法的説明（→ カテゴリ③）
- メンテナンス手順（→ カテゴリ④）
- 仕様情報（→ カテゴリ⑤）
- トラブル対応（→ カテゴリ⑥）

### 2.5 提供タイミング

- **初回使用時**: 機能を初めて使う時
- **操作開始時**: 操作を開始する時
- **操作ミス時**: 操作を誤った時
- **環境変化時**: 天気、温度、位置などが変化した時
- **警告灯点灯時**: 警告灯が点灯した時（操作ガイドとして）

### 2.6 提供方法

**主**: プロアクティブ通知（体験OS化）
- ARガイドの自動表示
- プッシュ通知
- 音声案内
- 画面表示

**補完**: オンデマンド検索（RAGシステム）
- ユーザーが質問した時に回答

### 2.7 具体例

| テキスト内容 | 提供タイミング | 提供方法 |
|------------|--------------|---------|
| 警告灯の見方 | 警告灯点灯時 | ARで説明表示、プッシュ通知 |
| エアコンの使い方 | 天気・温度に応じて | ARで操作ガイド、自動提案 |
| ワイパーの使い方 | 雨天時 | ARで操作ガイド、自動通知 |
| 給油の方法 | 給油口近く | ARで操作ガイド、自動通知 |
| カスタマイズ機能の使い方 | 設定画面を開いた時 | ARで操作ガイド |

### 2.8 2030年目標

- **自動化率**: 100%
- **提供精度**: 95%以上
- **ユーザー満足度**: 90%以上

## 3. カテゴリ②: リスク・危険・注意（Safetyモデル化）

### 3.1 定義

**リスク・危険・注意**とは、Safetyモデルに基づき、危険な状況や警告が必要な時に自動的に安全情報を提供する情報である。

### 3.2 技術モデル

**Safetyモデル（Safety Model）**
- 危険度の階層化（死亡リスク、重大傷害リスク、注意レベル）
- リアルタイム危険検知システム
- 優先度に応じた通知方法の選択

### 3.3 分類キーワード

#### 危険警告
- 危険、死亡、重大な傷害、事故、火災、感電
- やけど、破裂、横すべり、重大な傷害におよぶ
- 最悪の場合死亡、絶対にしない、禁止

#### 注意事項
- 注意、故障、損傷、おそれ、リスク、影響
- 悪影響、トラブル、異常、警告

#### 禁止事項
- しないでください、禁止、取り付けない、使用しない
- 貼り付けない、改造、使用しないでください

### 3.4 判断基準

✅ **このカテゴリに分類する**:
- 危険警告、安全注意事項
- 警告灯点灯時の対応（安全関連）
- 死亡・重大傷害のリスクがある情報
- リアルタイム通知が必要な情報
- 不正改造、禁止事項に関する情報

❌ **このカテゴリに分類しない**:
- 操作ガイド（→ カテゴリ①）
- 法的説明（→ カテゴリ③）
- トラブル対応（→ カテゴリ⑥）

### 3.5 提供タイミング

- **危険検知時**: 危険な状況を検知した時
- **警告灯点灯時**: 安全関連の警告灯が点灯した時
- **不適切な操作時**: 危険な操作を検知した時
- **メンテナンス時**: 危険な作業を行う時

### 3.6 提供方法

**主**: プロアクティブ通知（体験OS化）
- リアルタイム通知（優先度: 最高）
- ARで危険箇所を表示
- 音声警告
- 画面表示

**補完**: デジタル記録（法的証跡）
- 警告履歴の記録
- 改造履歴の記録

### 3.7 具体例

| テキスト内容 | 提供タイミング | 提供方法 |
|------------|--------------|---------|
| 警告灯点灯時の説明と対処法 | 警告灯点灯時 | リアルタイム通知、ARで説明 |
| 不正改造の警告 | 改造検知時 | 自動通知、デジタル記録 |
| 高電圧部位への接触警告 | メンテナンス時 | ARで危険箇所表示、自動通知 |
| 安全装置の警告灯が点灯しない場合の危険性 | 点検時 | 自動通知 |
| エンジンルームへの水かけの危険性 | 洗車時 | 自動通知、ARで注意事項表示 |

### 3.8 2030年目標

- **自動化率**: 100%
- **リアルタイム通知率**: 100%
- **事故削減率**: 50%以上

## 4. カテゴリ③: 法規制・コンプライアンス・説明責任（Regulatory & Obligation）

### 4.1 定義

**法規制・コンプライアンス・説明責任**とは、法的責任を果たすために記録・保存する必要がある情報である。

### 4.2 技術モデル

**Regulatory & Obligationシステム**
- 法的要件の明確化
- デジタル証跡の記録
- 透明性の確保（ユーザーが確認可能）

### 4.3 分類キーワード

#### 法的義務
- 法律で義務、義務付け、日常点検、定期点検、点検整備
- 国土交通省、届け出、不正改造、道路運送車両法

#### 法的説明
- データの記録、EDR、イベントデータレコーダー、開示
- 法的強制力、警察、裁判所、政府機関、訴訟
- T-Connect、利用規約、同意

#### 保証・免責事項
- 保証、保証範囲、保証いたしません、免責、責任

### 4.4 判断基準

✅ **このカテゴリに分類する**:
- 法的義務、法的説明
- 保証、免責事項
- データ記録、開示条件
- 証跡として記録が必要な情報
- メーカーの説明責任に関する情報

❌ **このカテゴリに分類しない**:
- 操作ガイド（→ カテゴリ①）
- 安全警告（→ カテゴリ②）
- 仕様情報（→ カテゴリ⑤）

### 4.5 提供タイミング

- **記録時**: データを記録する時
- **確認時**: ユーザーが確認する時
- **開示時**: 法的開示が必要な時
- **点検時期**: 法的義務の点検時期（通知も必要）

### 4.6 提供方法

**主**: デジタル記録（法的証跡）
- デジタルダッシュボード
- アプリ内記録
- クラウド記録
- 透明性の確保（ユーザーが確認可能）

**補完**: プロアクティブ通知（体験OS化）
- 点検時期の自動通知
- 重要な法的説明の通知

### 4.7 具体例

| テキスト内容 | 提供タイミング | 提供方法 |
|------------|--------------|---------|
| 車両データの記録と開示条件 | 記録時、確認時 | デジタル記録、ダッシュボードで確認可能 |
| EDR（イベントデータレコーダー）の説明 | 記録時、確認時 | デジタル記録、ダッシュボードで確認可能 |
| 保証範囲と保証期間 | 確認時、保証範囲判定時 | デジタル記録、アプリで確認可能、自動判定 |
| 日常点検整備の法的義務 | 点検時期 | デジタル記録 + プロアクティブ通知 |
| サイバー攻撃のリスク説明 | 取り付け時、確認時 | デジタル記録、取り付け履歴と連動 |

### 4.8 2030年目標

- **デジタル記録率**: 100%
- **法的責任の完全履行**: 100%
- **透明性の確保**: 100%

## 5. カテゴリ④: メンテナンス・点検指示（予兆保全モデル）

### 5.1 定義

**メンテナンス・点検指示**とは、予兆保全モデルに基づき、メンテナンス時期や点検が必要な時に自動通知する情報である。

### 5.2 技術モデル

**予兆保全モデル（Predictive Maintenance Model）**
- センサーデータによる異常検知
- 走行距離・期間による予測
- メンテナンス履歴の学習

### 5.3 分類キーワード

#### 定期メンテナンス
- エアコンフィルター、交換、15000km、12ヶ月、25000km
- タイヤローテーション、5000km、定期点検、定期メンテナンス

#### 日常点検
- タイヤ、空気圧、点検、ウォッシャー液、補充
- 外装、内装、手入れ、洗車、清掃

#### 部品交換
- 電池交換、電球、バルブ、ヒューズ、交換するには
- 取りはずす、取り付ける

### 5.4 判断基準

✅ **このカテゴリに分類する**:
- 定期メンテナンス、日常点検
- 走行距離、期間で判断できる情報
- センサーで検知できる情報
- 予兆保全モデルで予測できる情報
- 部品交換の手順

❌ **このカテゴリに分類しない**:
- 操作ガイド（→ カテゴリ①）
- 安全警告（→ カテゴリ②）
- 仕様情報（→ カテゴリ⑤）
- トラブル対応（→ カテゴリ⑥）

### 5.5 提供タイミング

- **走行距離到達時**: 指定の走行距離に到達した時
- **期間到達時**: 指定の期間が経過した時
- **センサー検知時**: センサーで異常を検知した時
- **メンテナンス開始時**: メンテナンスを開始する時

### 5.6 提供方法

**主**: プロアクティブ通知（体験OS化）
- 自動通知（走行距離、期間、センサー検知）
- ARガイドの自動表示
- ディーラー予約の自動提案

### 5.7 具体例

| テキスト内容 | 提供タイミング | 提供方法 |
|------------|--------------|---------|
| エアコンフィルター交換 | 15000kmごと、12ヶ月ごと | 自動通知、ARで交換手順ガイド |
| タイヤ空気圧点検 | センサー検知時 | 自動通知、ARで点検ガイド |
| ウォッシャー液補充 | 液面低下時 | 自動通知、ARで補充手順ガイド |
| タイヤローテーション | 5000kmごと | 自動通知、ディーラー予約を自動提案 |
| 電子キーの電池交換 | 電池残量低下時 | 自動通知、ARで交換手順ガイド |

### 5.8 2030年目標

- **自動化率**: 100%
- **予兆保全の精度**: 90%以上
- **メンテナンスコスト削減**: 30%以上

## 6. カテゴリ⑤: スペック・仕様・固有値（IDパーソナライズ）

- 必要なタイミングでの自動表示

### 6.3 分類キーワード

#### 容量・数量
- 容量、L、36L、3.3L、3.6L、4.8L、1.6L

- 0W-20、5W-30、推奨、適合、指定銘柄

#### 設定値
- 空気圧、kPa、kg/cm²、230、220、200、420
- トルク、N・m、kgf・cm、103、1050、ワット、W

### 6.4 判断基準

✅ **このカテゴリに分類する**:
- 容量、規格、設定値
- 車両固有の情報
- 参照が必要な情報
- パーソナライズされた情報
- 技術仕様

❌ **このカテゴリに分類しない**:
- 操作ガイド（→ カテゴリ①）
- 法的説明（→ カテゴリ③）
- メンテナンス手順（→ カテゴリ④）

### 6.5 提供タイミング

- **参照時**: ユーザーが参照する時
- **必要な時**: 給油時、交換時など、必要なタイミング
- **確認時**: ユーザーが確認する時

### 6.6 提供方法

**主**: デジタル記録（法的証跡）
- アプリ内データベース
- クラウド記録
- 確認可能なダッシュボード

**補完**: プロアクティブ通知（体験OS化）
- 必要な時に自動表示（給油時、交換時など）

### 6.7 具体例

## 7. カテゴリ⑥: 状態異常・トラブルシュート（状態遷移モデル）

### 7.1 定義

**状態異常・トラブルシュート**とは、状態遷移モデルに基づき、異常状態を検知して自動診断・対応方法を提示する情報である。

### 7.2 技術モデル

**状態遷移モデル（State Transition Model）**
- 正常状態 → 異常状態の遷移を検知
- 異常状態の分類（軽微、中程度、重大）
- 自動診断アルゴリズム

### 7.3 分類キーワード

#### 警告灯の意味と対応
- 警告灯、点灯、点灯時、対応、対処、消灯
- ブレーキ警告灯、高水温警告灯、エンジン警告灯
- SRSエアバッグ警告灯、ABS警告灯、PCS警告灯

#### よくある問題と解決方法
- 表示が更新されない、正しく表示されない、風量が減少
- 目詰まり、問題、解決、対処法

#### 緊急時の対応
- パンク、バッテリーあがり、救援、緊急、応急
- 故障、トラブル

### 7.4 判断基準

✅ **このカテゴリに分類する**:
- 警告灯の意味と対応
- よくある問題と解決方法
- 緊急時の対応
- 状態遷移モデルで検知できる情報
- 異常状態の診断と対応

❌ **このカテゴリに分類しない**:
- 操作ガイド（→ カテゴリ①）
- 安全警告（→ カテゴリ②）
- メンテナンス手順（→ カテゴリ④）

### 7.5 提供タイミング

- **異常検知時**: 異常状態を検知した時
- **警告灯点灯時**: 警告灯が点灯した時
- **問題発生時**: 問題が発生した時
- **エラー発生時**: エラーが発生した時

### 7.6 提供方法

**主**: プロアクティブ通知（体験OS化）
- 自動診断と通知
- ARガイドの自動表示
- 対処法の自動案内

### 7.7 具体例

| テキスト内容 | 提供タイミング | 提供方法 |
|------------|--------------|---------|

## 8. カテゴリ⑦: ライフスタイル・目的別利用（目的理解OS）

### 8.1 定義

**ライフスタイル・目的別利用**とは、ユーザーのライフスタイルや目的を理解し、最適な利用方法を提案する情報である。

### 8.2 技術モデル

**目的理解OS（Purpose Understanding OS）**
- ユーザーの行動パターン分析
- ライフスタイルの理解
- 目的に応じた情報提供

### 8.3 分類キーワード

#### ライフスタイル関連
- エコドライブ、長距離ドライブ、キャンプ
- 雪道、運転方法、活用方法
- 目的、用途、シーン

#### 目的別利用
- 省エネ、燃費向上、快適性
- 安全性、便利性、効率性

### 8.4 判断基準

✅ **このカテゴリに分類する**:
- ユーザーの目的に応じた提案
- ライフスタイルに応じた情報
- パーソナライズされた情報
- 目的理解OSで理解できる情報
- シーン別の利用方法

❌ **このカテゴリに分類しない**:
- 操作ガイド（→ カテゴリ①）
- 安全警告（→ カテゴリ②）
- メンテナンス手順（→ カテゴリ④）

### 8.5 提供タイミング

- **目的検知時**: ユーザーの目的を検知した時
- **シーン検知時**: 特定のシーンを検知した時
- **行動パターン検知時**: ユーザーの行動パターンを検知した時
- **質問時**: ユーザーが質問した時

### 8.6 提供方法

**主**: プロアクティブ通知（体験OS化）
- 目的に応じた自動提案
- ライフスタイルに応じた情報提供

**補完**: オンデマンド検索（RAGシステム）
- ユーザーが質問した時に回答

### 8.7 具体例

| テキスト内容 | 提供タイミング | 提供方法 |
|------------|--------------|---------|
| エコドライブの提案 | 運転パターンに応じて | 自動提案 |
| 長距離ドライブの準備 | 目的地・距離に応じて | 自動提案 |
| キャンプでの使い方 | 位置情報・天気に応じて | 自動提案 |
| 雪道での運転方法 | 天気・位置情報に応じて | 自動提案 |
| 機能の活用方法 | ユーザーの行動パターンに応じて | 自動提案 |

### 8.8 2030年目標

- **パーソナライズ率**: 90%以上
- **目的理解の精度**: 85%以上
- **ユーザー満足度**: 90%以上

## 9. 分類の実装ガイド

### 9.1 分類の優先順位

1. **カテゴリ②（リスク・危険・注意）**を最優先
   - 安全に関わる情報は最優先で分類

2. **カテゴリ③（法規制・コンプライアンス）**を次に優先
   - 法的責任に関わる情報は優先的に分類

3. **その他のカテゴリ**を状況に応じて分類

### 9.2 分類の判断フローチャート

```
テキストを分析
  ↓
危険・警告のキーワードがある？
  ├─ YES → カテゴリ②（リスク・危険・注意）
  └─ NO
      ↓
法的義務・説明のキーワードがある？
  ├─ YES → カテゴリ③（法規制・コンプライアンス）
  └─ NO
      ↓
メンテナンス・点検のキーワードがある？
  ├─ YES → カテゴリ④（メンテナンス・点検指示）
  └─ NO
      ↓
仕様・スペックのキーワードがある？
  ├─ YES → カテゴリ⑤（スペック・仕様・固有値）
  └─ NO
      ↓
トラブル・異常のキーワードがある？
  ├─ YES → カテゴリ⑥（状態異常・トラブルシュート）
  └─ NO
      ↓
操作・使い方のキーワードがある？
  ├─ YES → カテゴリ①（コンテキスト依存操作）
  └─ NO
      ↓
ライフスタイル・目的のキーワードがある？
  ├─ YES → カテゴリ⑦（ライフスタイル・目的別利用）
  └─ NO
      ↓
デフォルト: カテゴリ⑦（ライフスタイル・目的別利用）
```

### 9.3 分類の品質管理

#### 分類の一貫性チェック
- 同じセクション内のテキストは同じカテゴリに分類
- 親要素のカテゴリを子要素が継承
- セクションIDに基づく一貫性チェック

#### 分類の精度チェック
- キーワードマッチングの精度確認
- サンプルデータでの分類精度測定
- 人間によるレビュー
- 分類結果の統計分析

#### 分類の優先順位ルール
1. **カテゴリ②（リスク・危険・注意）**を最優先
   - 「死亡」「重大な傷害」「最悪の場合」などのキーワードがある場合
   - `attention_para`要素で危険警告が含まれる場合

2. **カテゴリ③（法規制・コンプライアンス）**を次に優先
   - 「法律で義務」「法的」「保証」「EDR」などのキーワードがある場合
   - `in01`セクションID（法的説明セクション）の場合

3. **カテゴリ④（メンテナンス・点検指示）**を次に優先
   - 「交換」「点検」「km」「ヶ月」などのキーワードがある場合
   - `ch06`セクションID（メンテナンスセクション）の場合

4. **その他のカテゴリ**を状況に応じて分類
