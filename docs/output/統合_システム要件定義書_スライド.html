📋 エグゼクティブサマリー

事業の目的

従来の「読む取扱説明書」を、AIエージェントを活用した"先回りして最適化した情報を届ける仕組み"へ進化させるDX事業。

核心コンセプト: 「探す」から「届く」へ

→ ユーザーが探さなくても、必要な情報が自動で届くシステムを構築

1. システム概要

解決する具体的な課題

  • 取扱説明書は読まれない：従来の「読む」前提の説明書では、実際の利用率が極めて低い
  • 機能の未活用：自動車機能の高度化により、ユーザーは多くの機能を十分に活用できていない
  • 情報が配布物として固定化されている：SDV（ソフトウェア定義車両）の普及により、継続的な更新が必要
  • 問い合わせ増加によるコスト増大：顧客満足度の低下や、問い合わせ増加によるコスト増大のリスク

2. 主な機能

コア機能（5個）

  1. RAGベースの検索型AI: 9つのインテント分類に対応（回答精度90-95%、レスポンス1秒以下）
  2. プロアクティブ情報提供: 緊急時自動案内、タイミングベース案内
  3. データクリーニング・構造化: PDFから構造化データへの変換
  4. マルチチャネル展開: スマホアプリ、Web版Chatbot、Web管理画面
  5. データ還流システム: ユーザーデータを商品企画へフィードバック

差別化機能（3個）

  1. DOI独自のRAG最適化技術
    • セマンティック境界チャンキング
    • マルチベクトル戦略
    • ハイブリッド検索（RRF）
    • 検索精度15-20%向上、検索速度40%向上
  2. インテント別動的最適化: 9つのインテント対応（検出精度95%+）
  3. Safe × Smart × Delightの3軸体験設計

2.3 付加価値機能（1-2個）

  1. 取扱データ編集のAI化
    • FR-1.1のデータクリーニング・構造化技術を活用した制作支援ツール

    • 高齢者・外国人・デジタルネイティブ向けカスタマイズ機能

3. 機能要件

FR-1: RAGベースの検索型AI

概要: 9つのインテント分類に対応した高精度な情報検索・回答システム

開発期間: 2-3ヶ月 | 優先度: 最高（コア機能）

FR-1: 詳細要件

  • FR-1.1 データクリーニング・構造化
    • PDFやドキュメントから不要な要素を除去
    • 取扱説明書特有の構造に基づいて構造化
    • 技術要件: PDFパーサー、構造解析アルゴリズム
  • FR-1.2 セマンティック境界チャンキング
    • 章・節・手順単位での分割
    • 目標: 検索精度15-20%向上
  • FR-1.3 マルチベクトル戦略
    • タイトル・本文・メタデータを個別にベクトル化
    • 目標: 検索速度40%向上

FR-1: 詳細要件（続き）

  • FR-1.4 ハイブリッド検索
    • ベクトル検索+キーワード検索の統合（RRF）
    • 検索前フィルタリング（コスト40%削減）
  • FR-1.5 インテント検出・ルーティング
    • 9つのインテント分類の検出（検出精度95%+）
    • インテント別に最適化された10種類以上のプロンプトテンプレート
  • FR-1.6 回答生成
    • 回答精度90-95%
    • レスポンス時間1秒以下
    • 複合質問対応

9つのインテント分類

検索型（5つ）

  1. 緊急時対応
  2. 操作方法の質問
  3. 段階的ガイド
  4. 機能比較・選択
  5. トラブルシューティング

プロアクティブ型（4つ）

  1. 状況認識による先回り案内
  2. タイミングベース案内
  3. 新機能・便利機能の発見提案
  4. 環境・状況に応じた最適提案

FR-2: プロアクティブ情報提供

概要: ユーザーが探さなくても、必要な情報が自動で届くシステム

開発期間: 2-3ヶ月 | 優先度: 最高（差別化機能）

Phase 1で実装すべき機能:

  • FR-2.1 緊急時自動案内: 警告灯点灯時、異常検知時に自動で対応方法を提示（目標: 緊急時問合せ削減率50%以上）
  • FR-2.2 タイミングベース案内: 納車直後、初回利用時など、特定タイミングで必要な情報を自動提示
  • FR-2.3 よくある質問の自動提示: ユーザーの状況に応じて、関連するよくある質問を自動表示

FR-2: Phase 1では実装しない機能（Phase 3以降）

  • FR-2.4 状況認識による先回り案内（Phase 3以降）
    • 車両状態・環境（天候・道路状況など）を認識し、必要な情報を自動提示
    • 例：雨天時のワイパー操作、高速道路進入時のクルーズコントロール案内
  • FR-2.5 新機能・便利機能の発見提案（Phase 4）
    • ユーザーが知らない便利な機能や使いこなし方を提案（Delight体験）
    • 機能発見率：20-30% → 60%以上
  • FR-2.6 環境・状況に応じた最適提案（Phase 4）
    • ユーザーの知識レベル・使用履歴・環境を総合的に判断し、最適な情報を提示（Smart体験）

FR-3: マルチチャネル展開

開発期間: 並行開発（Phase 1全体を通じて） | 優先度: 高

Phase 1で実装すべきチャネル:

  • FR-3.2 スマホアプリ
    • iOS/Android対応、プッシュ通知機能、AR機能
    • 技術スタック: Flutter
  • FR-3.3 Web版Chatbot
    • コールセンター・販売店スタッフ向け
    • 技術スタック: Next.js + TypeScript + Supabase
  • FR-3.4 Web管理画面
    • OEM向け管理画面、データ分析・レポート機能
    • 技術スタック: Next.js + TypeScript + Supabase

FR-3: Phase 1では実装しないチャネル（Phase 2以降）

  • FR-3.1 車載AIエージェント連携（Phase 2以降）
    • 車載システムとのAPI連携
    • リアルタイムでの情報提供
  • FR-3.5 コールセンター連携（Phase 2以降）
    • コールセンターシステムとの連携
    • 問合せ削減支援

FR-4: データ還流システム

概要: ユーザーの利用データや行動ログを商品企画へフィードバック

優先度: 中（Phase 2以降）

5.2 技術スタック選定の根拠

本格運用を考慮した選定方針

Phase 1（PoC・初期段階）

  • ベクトルDB: Supabase + pgvector（コスト効率、小規模向け）
  • モニタリング: LangSmith（無料プランから開始可能）
  • インフラ: Vercel + Supabase（サーバーレス、運用負荷低）

本格サービス化（Phase 2以降）

  • ベクトルDB: Pinecone（マネージド、99.99% SLA、自動スケーリング）
  • モニタリング: LangSmith + Datadog/New Relic（包括的監視）
  • インフラ: AWS/Azure/GCP（コンテナベース、Kubernetes対応）
  • セキュリティ: 包括的セキュリティスタック

推奨アプローチ（ハイブリッド・段階的移行）

RAGシステム（バックエンドAPI）

  • Phase 1: Python 3.12+ + FastAPI + LangChain/LangGraph
  • 本格運用: コンテナ化（Docker）→ Kubernetes（スケール時）
  • モニタリング: LangSmith（LLM監視）+ APM（アプリ監視）

データベース戦略

  • Phase 1: Supabase（PostgreSQL）+ pgvector（<10Mベクトル）
  • 本格運用: Pinecone（ベクトルDB）+ Supabase（PostgreSQL、アプリデータ）

Webアプリケーション

  • Web管理画面: Next.js 15+ + TypeScript + Supabase + Vercel
  • Web版Chatbot: Next.js 15+ + TypeScript + Supabase + Vercel
  • モバイルアプリ: Flutter 3.24+

選定理由（本格運用視点）

RAGシステム

  • Python + LangChain/LangGraph: 実績、コミュニティ、エージェント機能対応
  • FastAPI: 非同期処理、自動ドキュメント生成、本格運用向け機能
  • 最新LLM（GPT-4o/Claude 3.7/Gemini 2.0）: 高精度、最新機能

ベクトルDB

  • Pinecone（本格運用）: マネージド、99.99% SLA、自動スケーリング、サブ50ms p99レイテンシ
  • pgvector（PoC）: コスト効率、小規模向け、PostgreSQL統合

選定理由（本格運用視点）- 続き

モニタリング

  • LangSmith: LLMトレース、コスト追跡、RAG品質監視、本格運用必須
  • APM（Datadog等）: アプリケーションパフォーマンス、分散トレーシング

セキュリティ

  • 多層防御: WAF、認証・認可、シークレット管理、脆弱性スキャン
  • コンプライアンス: SOC 2、ISO 27001対応準備

CI/CD

  • GitHub Actions: 自動ビルド、テスト、デプロイ、本格運用必須
  • インフラ即コード: Terraform等でインフラのバージョン管理

既存の開発計画との整合性

  • 既存計画ではPython + FastAPI、Flutterが検討されている → 維持
  • 本格運用に向けて段階的に技術スタックを強化

  • マルチベクトル戦略

  • 継続的な品質改善プロセスの構築
