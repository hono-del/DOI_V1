## プロジェクト概要

### 目的
車両の100機能に対して、ユーザーの状況（位置・天気・時間・車両状態）を検知し、適切なタイミングでプロアクティブに機能ガイドを提供するシステムを構築する。

### 期待効果
- 問い合わせ件数：20〜40％削減
- ユーザー満足度：10〜15ポイント向上
- 機能利用率（ADAS等）：2〜3倍向上
- 新モデル開発リードタイム：10〜20％短縮

### 対象機能
100機能を以下9カテゴリに分類：
- 基本機能：14項目
- 安全機能：16項目
- 便利機能：21項目
- 高度機能：17項目
- エンジン・動力系：8項目
- メンテナンス・情報系：10項目
- セキュリティ：7項目
- 共通イベント：2項目
- その他：5項目

## Phase 2: プロアクティブ通知システム構築（3〜4ヶ月）

### 目標
リアルタイムデータを元に、適切なタイミングで機能ガイドを通知するシステムの構築

### 主要タスク

#### 2.1 データ収集基盤構築
- [ ] 車両データ収集ゲートウェイ開発
  - CANバスデータ取得モジュール
  - データサンプリング・バッファリング
  - セキュア通信（TLS/mTLS）
- [ ] 外部API統合
  - 天気API（OpenWeatherMap等）
  - 地図API（Google Maps等）
  - 道路情報API（VICS等）
- [ ] データストリーム処理基盤
  - Apache Kafka / AWS Kinesis
  - リアルタイムデータパイプライン

#### 2.2 イベント検知エンジン開発
- [ ] ルールエンジン実装（100機能×トリガー条件）
  - 優先度付け（嬉しい度1〜5に基づく）
  - 条件マッチング（複合条件対応）
  
**トリガー実装例:**

| 機能 | トリガー条件 | 実装ロジック |
|-----|------------|------------|
| ワイパー自動モード | 天気API=雨 AND ワイパー未使用 | `if weather.condition == "rain" and vehicle.wiper_status == "off" and not notified_today("wiper")` |
| レーンキープ | GPS=高速道路 AND 速度>60km/h AND 機能未使用 | `if location.road_type == "highway" and vehicle.speed > 60 and not feature_used("lane_keep", days=7)` |
| シートヒーター | 外気温<10℃ AND 季節初回 | `if weather.temp < 10 and season_changed() and not notified_this_season("seat_heater")` |

#### 2.3 通知優先度管理
- [ ] 通知スコアリングアルゴリズム
  ```python
  score = (
      urgency_score * 0.4 +      # 緊急度（安全>メンテ>便利）
      user_value_score * 0.3 +   # 嬉しい度（★1-5）
      timing_score * 0.2 +       # タイミング適合度
      frequency_penalty * 0.1    # 通知頻度ペナルティ
  )
  ```
- [ ] 通知頻度制御
  - 1日最大3件まで
  - 重要度高（★★★★★）は即時
  - 運転中は音声のみ、停車時は画面表示
- [ ] ユーザープリファレンス対応

#### 2.4 RAG連携
- [ ] トリガー→RAG検索フロー
- [ ] コンテキスト付き回答生成
- [ ] 根拠引用機能（取説ページ番号表示）

#### 2.5 通知配信システム
- [ ] プッシュ通知実装（Firebase / APNs）
- [ ] 車載ディスプレイ連携
- [ ] 音声合成（TTS）実装

### 成果物
- イベント検知エンジン
- 通知配信システム
- 管理ダッシュボード
- 運用マニュアル

### 要求事項

## リスク管理

### 技術リスク

| リスク | 影響度 | 発生確率 | 対策 |
|-------|-------|---------|------|
| CANデータ取得不可 | 高 | 中 | シミュレータ環境構築、代替データソース確保 |
| LLM API障害 | 高 | 低 | マルチプロバイダー対応、フォールバック実装 |
| 外部API制限 | 中 | 中 | キャッシング、レート制限管理、複数プロバイダー |
| スケーラビリティ不足 | 中 | 中 | 初期段階から水平スケール設計 |
| データ品質問題 | 中 | 高 | データバリデーション、異常検知 |

### ビジネスリスク

| リスク | 影響度 | 発生確率 | 対策 |
|-------|-------|---------|------|
| ユーザー受容性低下 | 高 | 中 | MVP早期リリース、ユーザーテスト実施 |
| 通知疲れ | 中 | 高 | 通知頻度制御、パーソナライズ |
| コスト超過 | 中 | 中 | 段階的リリース、コスト監視ダッシュボード |
| 競合出現 | 低 | 中 | 差別化機能（AR、パーソナライズ）強化 |

### セキュリティ・プライバシーリスク

| リスク | 影響度 | 発生確率 | 対策 |
|-------|-------|---------|------|
| 個人情報漏洩 | 高 | 低 | 暗号化、アクセス制御、監査ログ |
| 車両データ不正利用 | 高 | 低 | 匿名化、データ保持期間制限、同意管理 |
| なりすまし | 中 | 低 | 多要素認証、デバイス認証 |
| DDoS攻撃 | 中 | 中 | WAF、レート制限、CDN |

### 対策優先度
1. **高影響×高確率:** 即時対策必須
2. **高影響×中確率:** Phase 0で対策実装
3. **中影響×高確率:** Phase 1で対策実装
4. **その他:** 継続監視
