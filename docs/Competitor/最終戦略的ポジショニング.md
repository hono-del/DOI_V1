## 🎯 戦略的ポジショニングの3つの柱

### 1️⃣ **Digital Manual-first Experience Design**

#### 本質
- 取説 = 困りごとの最前線
- ここから始めることで、真のIntent/Journey設計が可能

#### 差別化
競合との違い：

| 競合                     | アプローチ         | 弱点                 |
|------------------------|----------------|---------------------|
| ServiceNow, Coveo      | CRM/検索から逆算   | 製品情報の構造理解がない |
| Siemens, PTC           | OEMから出発      | エンドユーザー視点が薄い |
| exaBase, Lazarus AI    | AIから出発       | 体験設計の知見がない    |
| **体験OS（自社）**       | **取説から出発**   | **困りごと起点のUX**  |

#### 具体的優位性
```
取説ソースデータの知見
  ↓
情報粒度・条件分岐・図版意味・手順ロジックを理解
  ↓
AIが理解できる形に変換
  ↓
Experience Ontology（体験構造データ層）
```

#### 重要な戦略的判断

**"唯一の全レイヤー横断"ではなく"最も広範な垂直統合"**

理由：
- ServiceNow, Siemens, Coveoも体験OS領域に進出している
- しかし彼らは「上から降りてきた」（CRM, PLM起点）
- 我々は「下から積み上げた」（製品情報起点）

👉 **この違いが、Experience Ontologyの質を決定する**

### 3️⃣ **Experience Ontology as a Core Asset**

#### 資産戦略の本質

```
案件型ビジネス         資産型ビジネス
     ↓          →          ↓
  個別最適化              構造資産化
     ↓                      ↓
  収益逓増なし          収益逓増あり
```

#### 積み上がる3つの資産

##### Asset 1: 製品体験データ層（Experience Ontology）

**内容**:
- 状態モデル（State Model）
- 故障パターン（Failure Patterns）
- 操作Intent（User Intent）
- ユーザージャーニー（Journey Map）
- canonical_term辞書（標準用語辞書）

**なぜ資産になるか**:
```
AIはモデルよりデータ構造が勝つ

LLMは汎用的 → どこでも使える
データ構造は個別 → 我々しか持てない
```

##### Asset 2: UX判断エンジン（体験OSコア）

**内容**:
- Next Best Action ロジック
- 観察点選定（T2レベル思考）
- ガイド生成ロジック

**ポイント**:
👉 **AIそのものではなく"判断設計"**

これは：
- OpenAIに依存しない
- 他社が真似できない（暗黙知の形式知化）
- 再利用可能

##### Asset 3: 情報 × 行動ログ

**蓄積メカニズム**:
```
取説DB → Digital Manual → 行動ログ
   ↓          ↓              ↓
 情報構造   実際の利用    困りごとパターン
   ↓          ↓              ↓
   ■━━━━━━Experience Ontology━━━━━━■
```

**これが可能にすること**:
- Push通知（予測的な情報提供）
- 予測ガイド（問題が起きる前にサポート）
- 状態理解（現在のコンテキスト把握）

## 🔥 競合との決定的な違い（WHY CMC?）

### 他社のポジションと弱点

#### 🏢 SIer（例: NTTデータ、日立など）
- **強み**: システム構築ができる
- **弱点**: 体験設計の知見がない
- **結果**: 機能は作れるが、UXが弱い

#### 🤖 AIベンダー（例: exaBase, Lazarus AI）
- **強み**: AIモデルが強い
- **弱点**: 製品知識・情報構造の理解がない
- **結果**: 汎用的すぎて製品特化の価値が出せない

#### 📚 CCMSベンダー（例: Paligo, QPP）
- **強み**: 文書管理ができる
- **弱点**: 行動理解、UX設計がない
- **結果**: 管理止まりで、体験に繋がらない

#### 🎯 体験系プラットフォーム（例: ServiceNow, Coveo）
- **強み**: エンタープライズ実績、行動分析
- **弱点**: 製品情報の構造化知見がない
- **結果**: 表層的な体験最適化に留まる

#### 🏭 製造系プラットフォーム（例: Siemens, PTC）
- **強み**: OEM業務理解、製品知識
- **弱点**: エンドユーザー体験の深堀りが弱い
- **結果**: OEM視点が強すぎる

### ✅ CMCのユニークポジション

```
製品情報 → 行動 → UX → AI判断
を一気通貫で設計できる会社
```

**具体的な強み**:

| 領域            | CMCの能力                          | 他社との差                       |
|---------------|---------------------------------|------------------------------|
| **情報構造**     | 取説ソースデータの構造理解              | SIer/AIベンダーにはない         |

| **体験設計**     | 困りごと起点のIntent/Journey設計      | SIer/製造系には弱い             |
| **AI活用**      | 判断設計（モデルではなくロジック）         | CCMS/製造系には ない            |
| **統合力**      | 制作〜体験OSまでの垂直統合             | **誰も持っていない**             |

## 💰 ビジネスモデルと収益構造

### 従来型（案件型）からの転換

#### Before: 案件型モデル
```
取説制作 → 納品 → 終了
   ↓
収益: 一回限り
資産: 蓄積されない
```

#### After: 資産型モデル
```
取説制作 → 構造化 → Experience Ontology → 体験OS
   ↓        ↓           ↓                  ↓
  初期    データ層     判断エンジン        継続収益
```

### 収益モデルの3層構造

#### Layer 1: Foundation（基盤収益）
- **サービス**: Digital Manual制作
- **価格**: 従来通り
- **目的**: 入口＋情報収集

#### Layer 2: Enhancement（拡張収益）
- **サービス**: 状況別ガイド、行動ナビ
- **価格**: 月額SaaS
- **目的**: 継続収益化

#### Layer 3: Platform（プラットフォーム収益）
- **サービス**: 体験OS基盤
- **価格**: エンタープライズライセンス＋従量課金
- **目的**: 資産の最大活用
